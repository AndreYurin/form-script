## Context

The server is a TypeScript ESM Express app that currently uses drizzle-orm for a small Postgres schema (projects, notices, script_runs). Persistence is accessed through a singleton `db` imported from `server/src/db/client.ts`, and queries are scattered across routes, runner modules, and the cron scheduler. SQL migrations live under `server/drizzle/` and are generated by `drizzle-kit`.

The app is pre-production — there is no live data or deployed instance. A single developer runs everything locally via Docker Compose + `npm run dev`. This means we can do a hard cutover without a data-migration dance.

The proposal (`proposal.md`) calls for replacing drizzle with MikroORM to get entity relations, unit-of-work semantics (useful for `runner/sync.ts` which upserts many notices at once), and a first-class migration workflow that scales with the schema.

## Goals / Non-Goals

**Goals:**
- Replace drizzle-orm with MikroORM PostgreSQL driver across the server.
- Model `Project`, `Notice`, `ScriptRun` as entity classes with explicit relations (`Project 1—n Notice`, `Project 1—n ScriptRun`, `Notice 1—n ScriptRun`).
- Provide a request-scoped `EntityManager` via middleware so each HTTP request / cron tick runs inside its own identity map.
- Replace `db:generate` / `migrate` scripts with MikroORM migrator equivalents and produce a single baseline migration matching the current schema.
- Preserve every existing HTTP API contract (routes and response shapes unchanged).

**Non-Goals:**
- No changes to the client (`client/**`).
- No changes to the legacy scraper scripts (`step1-collect-ids.js`, `step2-collect-details.js`) or the JSON-file bridge consumed by `runner/sync.ts`; only the DB write path inside `sync.ts` is rewritten.
- No production migration tooling or zero-downtime strategy — local DB is recreated.
- No switch to a different runtime validator; Zod (if used) stays as-is.
- No introduction of GraphQL, REST resource generators, or MikroORM's `@mikro-orm/nestjs` style decorators beyond what core requires.

## Decisions

### 1. MikroORM edition: reflect-metadata decorators

Use `@mikro-orm/core` with decorator-based entities and `reflect-metadata`, not the `EntitySchema` programmatic API.

**Why:** Decorator entities read closest to the existing drizzle schema file and give us TypeScript-native typing without a parallel schema definition. The project already builds with `tsx` + `tsc`, so enabling `experimentalDecorators` and `emitDecoratorMetadata` in `server/tsconfig.json` is a small change.

**Alternatives considered:**
- `EntitySchema` (no decorators): avoids tsconfig changes but produces verbose, duplicated type/runtime definitions. Rejected — worse DX for a small team.
- Prisma: more popular but requires a separate schema file and code-gen step, and doesn't give us unit-of-work semantics. Rejected — the proposal specifically calls for MikroORM.

### 2. Request-scoped EntityManager via middleware

Bootstrap a single `MikroORM` instance at server start. Expose the global `EntityManager` only for bootstrap work (seed, cron init). For HTTP requests, install an Express middleware that calls `RequestContext.create(orm.em, next)`, so every route handler gets a forked EM bound to the request lifecycle.

**Why:** MikroORM's identity map is not safe to share across concurrent requests. `RequestContext` is the documented, low-friction way to isolate EMs per request. Cron ticks get the same treatment via an explicit `await orm.em.fork().transactional(...)` wrapper.

**Alternatives considered:**
- Pass `em` explicitly through function arguments: maximally explicit but requires threading `em` through every runner and cron function. Rejected — too much churn for a local tool.
- Share the global EM: simplest but corrupts identity map under concurrent load. Rejected outright.

### 3. Baseline migration only; drop and recreate local DB

Generate one baseline MikroORM migration that recreates the current schema (projects, notices, script_runs, enums, unique `(project_id, notice_id)`), and delete the `server/drizzle/` directory. Developers run `docker compose down -v` then `docker compose up -d` and `npm run --workspace server migrate` once.

**Why:** No production data exists. Trying to translate drizzle migration history into MikroORM migration history has zero payoff and real risk of drift. A clean baseline is the smallest correct step.

**Alternatives considered:**
- Port each drizzle migration 1:1 into a MikroORM migration: preserves history, but history has no consumers. Rejected.
- Use `schema:update` at startup instead of migrations: fine for a toy, but the proposal explicitly wants a real migration workflow. Rejected.

### 4. Unit-of-work for `runner/sync.ts`

Rewrite the JSON→DB sync as a single transaction: fork an EM, `em.find` existing notices for the project by `noticeId`, upsert via entity mutation or `em.upsert`, then one `em.flush()`. Keep the skip rules (don't overwrite `rejected` or `details_collected`) as in-memory checks before mutating entities.

**Why:** The current drizzle implementation does one query per notice. MikroORM's unit-of-work lets us batch this into one flush with full identity-map dedupe, which is exactly the kind of code path that motivated the switch.

**Alternatives considered:**
- Keep per-row upserts: functionally fine but wastes the main win of the migration. Rejected.

### 5. Enums stay as string literal types + `@Enum`

Use `@Enum({ items: () => NoticeStatus })` with a TS string enum for `notice.status` and `script_runs.status`. Map to Postgres `text` + CHECK constraint via MikroORM's native enum handling, matching the current drizzle behavior.

**Why:** Keeps the Postgres-level representation identical to today, so queries and existing JSON data continue to work unchanged.

### 6. CLI config file

Add `server/src/mikro-orm.config.ts` exporting the `Options` object (entities, driver, migrations path, seeder path, debug). Point `@mikro-orm/cli` at it via the `"mikro-orm"` key in `server/package.json`.

**Why:** One config shared between runtime bootstrap and CLI (`migration:create`, `migration:up`, `seeder:run`) avoids drift between dev commands and the running server.

## Risks / Trade-offs

- **Decorator metadata requires `experimentalDecorators` + `emitDecoratorMetadata`** → Enable in `server/tsconfig.json` and import `reflect-metadata` once at the top of `server/src/index.ts`. Verified to work with `tsx` in dev and `tsc` for build.
- **`RequestContext` middleware must wrap all routes, not just some** → Mount it before the API router in `server/src/index.ts`; add a smoke test (or at least a manual check) that a route reading `em` works after the switch.
- **Cron jobs run outside Express request lifecycle** → Explicitly fork an EM inside each cron callback (`const em = orm.em.fork(); await em.transactional(...)`). Without this, cron writes corrupt the identity map.
- **`em.upsert` semantics differ from drizzle's `onConflictDoUpdate`** → Validate against the `(project_id, notice_id)` unique constraint and the "don't clobber rejected/details_collected" rule by running `db:migrate-json` against a fresh DB and diffing against a pre-migration snapshot.
- **MikroORM bundle size / cold start** is larger than drizzle → Acceptable for a local admin tool; no deployment cold-start budget to protect.
- **Decorator imports can trip circular dependencies** between entities → Use string-based relation targets (`@ManyToOne(() => Project)`) and split entities into one file per class under `server/src/db/entities/`.
- **Rollback story**: the git commit before this change is the rollback. Because the DB is recreated, reverting the code and running `docker compose down -v && up -d && migrate` returns to the drizzle world.

## Migration Plan

1. Add MikroORM dependencies; remove drizzle dependencies after the cutover compiles.
2. Create `server/src/db/entities/{project,notice,script-run}.ts` and `server/src/db/enums.ts`.
3. Create `server/src/mikro-orm.config.ts` and wire `"mikro-orm"` config into `server/package.json`.
4. Replace `server/src/db/client.ts` with a `MikroORM.init()` bootstrap + exported `orm` / `em`.
5. Install `RequestContext` middleware in `server/src/index.ts`; import `reflect-metadata` at the top.
6. Port each route and runner module to use `req` / forked EMs. Do this file-by-file with the TypeScript compiler as the safety net.
7. Generate the baseline migration (`mikro-orm migration:create --initial`), delete `server/drizzle/` and `server/drizzle.config.ts`.
8. Rewrite `server/src/db/seed.ts` and `server/scripts/migrate-json-to-db.ts`.
9. Update `server/package.json` scripts and `CLAUDE.md` Quick Start.
10. Local verification: `docker compose down -v && up -d`, `npm run --workspace server migrate`, `db:seed`, `db:migrate-json`, `npm run dev`, click through the dashboard, trigger step1 + step2, verify script_runs + notices populate correctly.

## Open Questions

- Do we want `@mikro-orm/seeder` for `seed.ts`, or keep the current "plain function called from a script" style? Leaning toward keeping the plain function — less machinery.
- Should `ScriptRun.log` stay as `text` or become a relation to a separate `script_run_logs` table? Out of scope for this change; note for a follow-up.
- Do we need a CI check that entities match the last migration (`mikro-orm migration:check`)? Probably yes eventually; out of scope here.
